demand-index

This benchmark is authored as MeTTa data, not as a rho process. The source
contains two tables: (watch topic sink) subscriptions and (fact topic payload)
facts. MeTTa recursively compiles those tables into rho receives and sends,
then emits the resulting rho process as data. The benchmark runner executes
that generated process through `--lang rhocalc` and through rholang-cli.

This models a demand/fact notification frontier: many watchers may demand
several topics, and several facts may arrive on those topics. --lang rhocalc
can run the compiled process, but it cannot itself interpret the two MeTTa
tables and synthesize the rho network.

Files:
- sizes.tsv: default topic/watch/fact sizes for this benchmark only.
- generate.sh SIZE OUT_DIR: writes demand_index_SIZE.metta,
  demand_index_SIZE.mrho, and demand_index_SIZE.rho.
- run.sh: runs only demand-index against CeTTa --lang rhocalc and
  f1r3node rholang-cli.

The generated `.mrho` file is CeTTa's strict-core rho concrete syntax; the
generated `.rho` file is the surface rho/Rholang spelling used for
cross-engine runs.

The default ladder starts at 32/48/64 now that the evaluator visible-variable
cap has been removed. Larger SIZES values may investigate MeTTa builder limits
or true rho runtime reduction cost rather than pure generator overhead.

CeTTa prints the residual process after deterministic reduction;
rholang-cli prints a deploy-cost summary of roughly 70 bytes. The measured
wall time therefore includes asymmetric output-formatting cost. Treat these
rows as engine-end-to-end measurements, not scheduler-to-scheduler
measurements.
