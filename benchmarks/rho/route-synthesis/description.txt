route-synthesis

This benchmark is authored as MeTTa data, not as a rho process. The source
contains a route table of (edge src dst) facts and MeTTa builder rules that
synthesize a strict-core rho forwarding network and emit it as data. The
benchmark runner executes the generated process through `--lang rhocalc` and
through rholang-cli.

That is the point of the benchmark: --lang rhocalc can run the generated rho
process, but it cannot itself consume the route-table data and construct the
process. The Rholang lane receives the equivalent generated network for
cross-engine timing.

Files:
- sizes.tsv: default route-table sizes for this benchmark only.
- generate.sh SIZE OUT_DIR: writes route_synthesis_SIZE.metta,
  route_synthesis_SIZE.mrho, and route_synthesis_SIZE.rho.
- run.sh: runs only route-synthesis against CeTTa --lang rhocalc and
  f1r3node rholang-cli.

The generated `.mrho` file is CeTTa's strict-core rho concrete syntax; the
generated `.rho` file is the surface rho/Rholang spelling used for
cross-engine runs.

The default ladder starts at 32/48/64 now that the evaluator visible-variable
cap has been removed. Larger SIZES values may investigate MeTTa builder limits
or true rho runtime reduction cost rather than pure generator overhead.

CeTTa prints the residual process after deterministic reduction;
rholang-cli prints a deploy-cost summary of roughly 70 bytes. The measured
wall time therefore includes asymmetric output-formatting cost. Treat these
rows as engine-end-to-end measurements, not scheduler-to-scheduler
measurements.
